## Decisions

- **Auth scope reduced from "every proxied path" to "/chat only" mid-execution.** The plan called for auth on `/chat`, `/assets`, `/api`, `/favicon-logo.svg`. Gate 4 revealed the SPA's streaming agent POST (`POST /api/.../threads/<id>/agent`) omits Basic Auth credentials from the request — the browser's stored Basic Auth doesn't reach it. nginx 401'd, browser re-prompted, user re-typed password, fetch still omitted creds → loop. Moved auth to `/chat`-only entry gate; internal SPA paths open. Trade-off: anyone who discovers `/api/*` URL patterns could query the agent without auth (API-token-burn risk, no data exposure). Mitigated by Anthropic spend cap + URL not widely shared + Tailnet `:3000` as the project-team path. Documented in the limitation entry and ARCHITECTURE.
- **Credential surfaced once in chat with rotation instructions, scratch file deleted on confirmation.** Chat transcript captured the password once; Gordon saved to password manager; `scratch/basic_auth_password.txt` deleted. Rotation procedure (`sudo htpasswd -B /etc/nginx/.htpasswd analyst` without `-c`) documented in SETUP.md §14 and the limitation entry.
- **Added branding-asset regex location** (`/oxygen-*.{svg,gif,png}`) after the initial gate-4 attempt showed 404s for `/oxygen-dark.svg`, `/oxygen-light.svg`, `/oxygen-loading-dark.gif`. The SPA loads these at runtime; without proxy locations they fell through to the portal's `try_files`. Regex location catches the whole family.
- **`/home` and `/threads` added as proxy locations** for SPA client-side routing targets. The SPA navigates to `/home` and `/threads/<id>` from inside the workspace; without proxy locations they'd 404. Both proxy to `:3000` unauth.

## Issues encountered

### SPA's streaming agent POST omits Basic Auth credentials

First gate-4 attempt: Gordon walked the portal → `/chat` → auth prompt → SPA loaded → asked a question → got a second auth prompt that couldn't be passed even with the correct password. Access log showed `POST /api/.../threads/<id>/agent` with `-` in the auth-user field (no Authorization header sent). Other `/api/*` GETs were fine (`analyst` in user field). The streaming POST is the SPA's "run agent" call — likely using fetch with `credentials: 'omit'` (Server-Sent Events / streaming response style).

Browsers don't auto-include Basic Auth on requests where the fetch suppresses credentials. The user typing the password into the prompt doesn't help — the SPA's code is the gating layer. Fix: remove auth from `/api` (and other SPA-internal paths). Documented as a discovered constraint in the limitation entry.

### Stale browser cache after the auth scope change

After redeploying with auth scoped to `/chat` only, Gordon's browser still showed the auth dialog because it had cached the failed-auth state from the prior config. Fresh tab + reload resolved it. Access log timestamps confirmed: the 401s in the screenshot were ~01:36-01:37 UTC, the revised config went live at 01:38, the verification curls at 01:39 returned 200.

### Initial trust-surface design assumed auth on every path

The plan brief was written from a "defense in depth" angle — auth on every proxied path so the JS bundle couldn't be fetched anonymously. Reality: the SPA's auth-credential-omission means defense-in-depth at the nginx layer is impossible without changing the SPA's fetch code. The pragmatic alternative (entry-gate-only auth + Anthropic spend cap + URL not widely shared) is the throwaway-scaffolding-shaped fit for MVP 1.5.
