## Key findings worth knowing

1. **Builder Agent CLI works in `--local` mode.** `oxy agentic run
   --domain builder builder --question "..."` end-to-end:
   `read_file`, `propose_change` with HITL Accept/Reject via `oxy
   agentic answer`, `execute_sql`, multi-turn solving. STACK.md §1.8
   already named the CLI domain; Plan 11's design didn't. The
   opportunistic principle's pre-flight saved this plan from a
   stop-and-surface.

2. **Builder CLI token-budget resume hangs.** `oxy agentic answer
   --answer "Continue with double budget" <RUN_ID>` accepts the
   answer but the run-state-machine doesn't resume. Verified
   empirically; logged in `docs/limitations/plan-11-builder-cli-token-budget-hang.md`
   as customer-feedback for Oxy. Workaround used in Plan 11: cancel
   the run, hand-write the file with Builder's verbatim SQL.

3. **STACK.md §1.9 corrected.** `oxy run apps/<name>.app.yml` is NOT
   supported in oxy 0.5.47 — rendering is SPA-only. Builder doesn't
   use a dedicated `run_app` tool either; it runs each task's SQL via
   `execute_sql`. Both findings logged.

4. **Crime data multi-code groupings.** The Plan 13 design's "1 row"
   assumption was off by 2,874. Two structural NIBRS conventions are
   responsible, not data quality drift. Plan 13 deviated to keep
   `offense_code` populated (cleaner analyst surface).

5. **Daily `partial` runs root-caused.** The drift-fail since Session
   31 was a current-year baseline + tight tolerance issue, not a
   real data anomaly. Plan 14a's surgical fix + model update means
   future current-year rows auto-deactivate.

6. **Rat-complaints filter shape.** `LIKE '%rat%'` catches false
   positives in `consideration` / `registration`. The correct filter
   is an explicit 4-type IN-list + `TRIM()` (handles `School Rodent
   Control ` trailing-space padding, same family as the existing
   `block-code-padded` pattern). 14,036 rows total.
