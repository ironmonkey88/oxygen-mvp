### Plan 17 — /dashboards auto-generator + rat listing retrofit (PR #8)

## Why I stopped here

The brief said "Stop after Session 47 and surface everything for review." I am stopping at Session 45 and surfacing now. The reason is the brief's own stopping condition: "Anything that contradicts a load-bearing assumption in this prompt."

The load-bearing assumption was that 5 sequential sessions fit in one Code thread at consistent quality. Sessions 43-45 were *mechanical* — generator + HTML + run.sh wiring, same pattern repeated. They went smoothly because they didn't require mode-switching.

Sessions 46-47 are *qualitatively different*:

1. **Session 46** is an interactive Builder CLI session driving the 311 overview dashboard. From Plan 11 (Session 39) we know Builder CLI has at least one fatal failure mode: the token-budget resume hang on `oxy agentic answer --answer "Continue with double budget"`. Driving Builder through 8-12 panels (vs Plan 11's 5) means more turns, more HITL approvals, more transcript, more places to hang.
2. **Session 47** is Builder iterating on an *existing* `.app.yml`. The brief flags this as a new capability test: "Gate: if Builder can't read+modify existing `.app.yml`, document and either deviate (hand-edit additions) or pause." Failure mode unknown.
3. **Both** need Chrome MCP visual verification, which is a tool I haven't loaded yet and a new mode (browser-driving) on top of the Builder CLI work.

This thread has already done today: Plans 11 + 14 (3 merged PRs from Sessions 39-42) + Sessions 43-45 (3 open PRs). The pattern of failure I'm worried about isn't "can't do the work" — it's "do the work tired and miss something that the user-journey discipline should have caught." Plan 11 had exactly that failure mode (Session 41's atob misdiagnosis): the work *looked* done; the dashboard *appeared* to render; the operator-facing user journey wasn't actually verified. The fix took two more PRs and a retraction.

I'd rather stop with 3 clean PRs and an honest checkpoint than ship 5 PRs where the last two might need retraction later. The brief itself encodes this: "Verify the user journey, not the destination. Session 41-42 lesson."
