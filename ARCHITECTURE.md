# ARCHITECTURE.md — Oxygen MVP

## Stack Decisions

| Layer | Tool | Why |
|---|---|---|
| Ingestion | **dlt** | Python-native, declarative, mature REST API support. Chosen over Oxygen's native Airway. |
| Warehouse | **DuckDB** | Zero-config embedded OLAP. Sufficient for 1.17M total records (~100-115k/year). |
| Transformation | **dbt Core** | Gordon knows dbt deeply. Airform (Oxygen's native option) only added April 2026 — too new. |
| Semantic Layer | **Airlayer** | Oxygen-native `.sem.yml`. Source of truth for all agent queries. |
| Q&A | **Answer Agent** | Standard Oxygen `.agent.yml` with `execute_sql` tool. Oxygen's term for a data chat agent. |
| Routing | **Routing Agent** | `type: routing` agent that dispatches to answer agents. Added in MVP 4 only. |
| Dashboards | **Airapp** | Declarative `.app.yml` — Oxygen's equivalent of a Looker dashboard. |
| Builder | **Builder Agent** | Oxygen IDE copilot. Not configured — it's a UI feature used interactively. |
| Deployment | **AWS EC2** | Single instance, internal use first. Ubuntu 24.04 LTS ARM. |
| LLM | **Claude Sonnet 4.6** | Best price/performance for analytics Q&A. Via Anthropic API. |

## Data Flow: Medallion Architecture

| Layer | dbt Model | What Happens |
|---|---|---|
| Bronze | `raw_311_requests.sql` | Exact mirror of SODA API response. No transformations. Arrival checks only. |
| Silver | `stg_311_requests.sql` | Normalize field names, cast types, deduplicate, **redact PII**. |
| Gold | `fct_311_requests.sql` | Business logic, aggregations, final fact table for analytics. Location fields denormalized. |

Airlayer and agents point to **Gold layer only**.

## Database Schema Design

### Schemas

| Schema | Purpose |
|---|---|
| `bronze` | Raw source data — exact mirror of API, no transforms |
| `silver` | Cleaned, typed, deduped, PII redacted (MVP 3) |
| `gold` | Business-ready facts and dimensions |
| `admin` | Infrastructure — profiling, DQ tests, pipeline metadata |

### Tables

| Schema | Table | Description |
|---|---|---|
| `bronze` | `raw_311_requests` | Exact mirror of Somerville SODA API response |
| `bronze` | `raw_dbt_results` | Raw load of dbt `run_results.json` after each run |
| `silver` | `stg_311_requests` | Normalized, typed, deduped, PII redacted |
| `gold` | `fct_311_requests` | Fact table — one row per 311 request, location denormalized |
| `gold` | `dim_date` | Standard date spine, generated by dbt macro |
| `gold` | `dim_request_type` | Request type dimension |
| `gold` | `dim_status` | Request status dimension |
| `admin` | `fct_data_profile` | Column-level profiling snapshots — observational only |
| `admin` | `dim_data_quality_test` | One row per defined test — baselines and dbt tests |
| `admin` | `fct_test_run` | One row per test per run — pass/fail results |

> Location fields (ward, neighborhood, lat/long) are denormalized onto `fct_311_requests` for MVP 1. Promoted to `dim_location` in MVP 3.

## Admin Schema: Table Designs

### `admin.fct_data_profile`
Observational only. Runs every pipeline execution. Does not generate rows in `fct_test_run`.

| Column | Type | Description |
|---|---|---|
| `profiled_at` | timestamp | When the profile was run |
| `table_name` | varchar | e.g. `raw_311_requests` |
| `column_name` | varchar | e.g. `request_type` |
| `row_count` | integer | Total rows in the table |
| `null_count` | integer | Nulls in this column |
| `pct_null` | float | null_count / row_count |
| `distinct_count` | integer | Distinct non-null values |
| `pct_distinct` | float | distinct_count / row_count |
| `min_value` | varchar | Min value cast to varchar |
| `max_value` | varchar | Max value cast to varchar |
| `min_length` | integer | Shortest string length |
| `max_length` | integer | Longest string length |
| `avg_length` | float | Average string length |

### `admin.dim_data_quality_test`
One row per defined test. Covers baseline comparisons and dbt tests only — not profiling runs.

| Column | Type | Description |
|---|---|---|
| `test_sk` | integer | Surrogate key |
| `test_id` | varchar | Natural key — e.g. `baseline.raw_311_requests.year_2016.row_count` |
| `test_type` | varchar | `baseline`, `dbt_generic`, `dbt_singular` |
| `table_name` | varchar | Table being tested |
| `column_name` | varchar | Column being tested — null if table-level |
| `metric` | varchar | What's being measured — `row_count`, `pct_null`, `unique`, `not_null` etc. |
| `grain` | varchar | What slice — e.g. `year=2016`, `all` |
| `expected_value` | varchar | The certified known-good value. For dbt tests: `0` (zero failures) |
| `tolerance_pct` | float | Acceptable drift % before fail — e.g. `0.01` for 1% |
| `is_active` | boolean | Soft-delete — deactivate without losing history |
| `certified_at` | timestamp | When this baseline was signed off |
| `certified_by` | varchar | `system` for auto-generated, `gordon` for manually certified |

### `admin.fct_test_run`
One row per test per dbt run. Sources from both baseline comparisons and parsed dbt test results.

| Column | Type | Description |
|---|---|---|
| `test_run_sk` | integer | Surrogate key |
| `test_sk` | integer | FK to `dim_data_quality_test` |
| `run_id` | varchar | dbt run identifier |
| `run_at` | timestamp | When this run executed |
| `actual_value` | varchar | What the data showed |
| `expected_value` | varchar | Copied from dim at time of run |
| `variance_pct` | float | Actual vs expected % difference |
| `status` | varchar | `pass`, `fail`, `warn` |
| `failure_message` | varchar | Human-readable explanation on fail |

## Data Quality Design

Three separate concerns — do not mix them:

| Concern | Table | Assertional? | Fails run? |
|---|---|---|---|
| Profiling | `fct_data_profile` | No — observational only | Never |
| Baseline comparisons | `fct_test_run` | Yes | Yes, on drift beyond tolerance |
| dbt tests | `fct_test_run` | Yes | Yes, on failure |

Baselines are auto-generated on first run with `certified_by = 'system'`. Manually promote to `certified_by = 'gordon'` to certify.

## Run Order

**Always use `run.sh` — never run steps manually.**

```bash
#!/bin/bash
set -e

# 1. Ingest source data
python dlt/somerville_311_pipeline.py

# 2. Transform
dbt run --project-dir dbt

# 3. Test (capture exit code but don't halt — we want to record failures)
dbt test --project-dir dbt || true

# 4. Capture dbt test results
python dlt/load_dbt_results.py

# 5. Parse results into admin schema
dbt run --select admin --project-dir dbt
```

`run.sh` in the project root is the single entry point. Never run dlt, dbt, or oxy commands individually.

If you see a `database is locked` error, check for competing processes:

```bash
lsof data/somerville.duckdb
```

## Oxygen Component Reference

### Airlayer (Semantic Layer)
- Equivalent to: dbt metrics + LookML
- File type: `.sem.yml`
- Defines: views, dimensions, measures, topics
- Topics group views by business domain (like Looker Explore)
- Docs: https://oxy.tech/docs/guide/learn-about-oxy/semantic-layer.md

### Answer Agent
- Equivalent to: Looker's "Ask" / a Looker chatbot
- File type: `.agent.yml`
- Key tool: `execute_sql` — agent writes and runs SQL against DuckDB
- Gets schema context from Airlayer via `context` block
- Docs: https://oxy.tech/docs/guide/learn-about-oxy/agents.md

### Routing Agent
- Equivalent to: a dispatcher / intent classifier
- File type: `.agent.yml` with `type: routing`
- Uses vector embeddings to match user queries to the right answer agent or workflow
- Added in MVP 4 — not needed until metric library is built out
- Docs: https://oxy.tech/docs/guide/learn-about-oxy/routing-agents.md

### Airapp (Dashboards)
- Equivalent to: Looker dashboards
- File type: `.app.yml`
- Declarative config — not drag-and-drop
- Docs: https://oxy.tech/docs/guide/learn-about-oxy/data-apps.md

### Builder Agent
- Oxygen's IDE copilot for building and editing project files
- Not configured via YAML — used interactively in the Oxygen UI
- No setup required

## ⚠️ DuckDB File Locking

DuckDB allows only **one writer at a time**. All three tools share `data/somerville.duckdb`:

- **dlt** writes during ingestion
- **dbt** writes during transformation
- **Oxygen agents** read (and occasionally write) during query execution

**Always run via `run.sh` — never run steps concurrently.**

## Deployment Architecture

- **Platform:** AWS EC2, Ubuntu 24.04 LTS ARM
- **Instance:** `t4g.medium` (ARM, 2 vCPU, 4 GB RAM) — suitable for 1–3 internal users
- **Oxygen port:** 3000 (HTTP)
- **Security group:** SSH (22) locked to Gordon's IP; port 3000 open to all
- **Outbound:** Must reach `data.somervillema.gov` (SODA API) and `api.anthropic.com`
- **Process management:** Run Oxygen as a systemd service (see SETUP.md)

### Future (not in scope for MVP)
- HTTPS via Caddy or nginx
- Multi-workspace cloud mode for team access + GitHub integration
- Amazon RDS for PostgreSQL (replace Docker-managed Postgres)
- Tailscale for access control (planned for MVP 3)
- `oxy serve` instead of `oxy start` for production

## Data Source

**Somerville 311 Open Data API**
- Portal: https://data.somervillema.gov
- Dataset ID: `4pyi-uqq6`
- Standard: Socrata Open Data API (SODA)
- Volume: ~100-115k requests/year (1.17M total, 2015–2026)
- Auth: None required
- Base URL: `https://data.somervillema.gov/resource/4pyi-uqq6.json`

SODA query patterns:
```
?$limit=50000&$offset=0
?$where=date_created >= '2024-01-01 00:00:00' AND date_created < '2025-01-01 00:00:00'
?$order=id ASC
/api/views/4pyi-uqq6.json       # schema
/api/views/metadata/v1           # discover datasets
```
